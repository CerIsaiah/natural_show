Control Loop

Hey, I'm Isaiah Cerven

I put this site and demo together for Natural as a concrete example of how I think about payments, agents, and shipping a usable front end.

About me

I've been an entrepreneur for as long as I can remember. I've worked with agencies on B2B projects, and I built my own consumer startup and grew it to 3,000+ users. That mix — selling to businesses, running a product people actually use — shaped how I think about product and execution.

For the past year I've been at Mastercard, in Networks clearing, where I work on end-to-end payment processing — how money actually moves through the network. Sitting in that world, what Natural is building feels especially relevant: the same primitives I see every day (limits, risk, trust, settlement) are exactly what agentic payments need, just redesigned for software that acts on its own.

Control Loop is an interactive walkthrough: human path first, then a clearly labeled agent section for the same rails with software layers, narrated chapter by chapter. I built it so there is a single link to share instead of repeating the same deck.

Why this matters for payments

Networks and issuers already evaluate rich signals on each transaction. Software that initiates spend still needs clear authorization, safe retries, and a trail that Finance or compliance can reconstruct. The demo states those requirements side by side with the human path so the gap is visible.

What I would prioritize and what I am still thinking through

Based on Networks clearing at Mastercard. Working directions, not a product commitment.

  • Reconciliation artifacts, not only successful auth. I would want anything an agent emits to stay legible when batch files are matched days later. Still weighing what minimum field set belongs in clearing-grade output so agent spend does not leave orphan auths.
  • Deterministic policy in front of probabilistic models. I would put repeatable rules in front of autonomous wallet spend so investigators get the same story on replay. The split between network scoring and issuer policy is something I would want explicit in any audit story.
  • Data on the formal settlement path. I would push intent and policy references into what operations actually load, not parallel chat. Scheme messages stay cramped, so I am interested in whether richer rails—ISO 20022 is the usual example—could carry agent lineage without bolting shadow databases.
  • Issuance aligned with policy. I would bind virtual PAN creation to the signed approval so caps do not drift from tickets. Whether issuers expose that as an API others can adopt, or it stays bespoke per program, is an industry-shape bet more than a solo call.
